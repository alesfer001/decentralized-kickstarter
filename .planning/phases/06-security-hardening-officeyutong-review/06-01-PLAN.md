Two changes to the merge path:

1. **Reject merge after deadline:** In the `if !is_after_deadline` section (line 321-324), merge is currently only allowed when `is_after_deadline == false`. This is already correct — merge is rejected after deadline because the code falls through to the release/refund path. But verify this is the case. If since_raw == 0 and actual block is past deadline, the merge still proceeds (this is the Issue 4 concern). However, since we can't enforce "before deadline" on CKB (since is a floor, not ceiling), the pragmatic fix is to document this limitation. The merge output retains the same lock, so funds remain locked regardless.

   Add a comment documenting this:
   ```rust
   if !is_after_deadline {
       // BEFORE DEADLINE (since == 0): only merge is allowed
       // Note: since=0 doesn't prove actual block < deadline, but merge output
       // retains the same lock script, so funds remain locked regardless.
       return validate_merge(&lock_args);
   }
   ```

2. **Assert identical lock args on merge inputs:** In `validate_merge()`, after getting `our_lock_hash`, add explicit lock args comparison for all group inputs:
   ```rust
   // Verify all group inputs have identical lock args (defense in depth)
   // Lock hash equality already implies this, but explicit check adds clarity
   let our_lock_args = match load_script() {
       Ok(s) => s.args().raw_data().to_vec(),
       Err(_) => return ERROR_INVALID_ARGS,
   };
   
   for i in 0.. {
       match load_cell_lock(i, Source::GroupInput) {
           Ok(lock) => {
               if lock.args().raw_data().as_ref() != our_lock_args.as_slice() {
                   debug!("Merge: input {} has different lock args", i);
                   return ERROR_MERGE_LOCK_MISMATCH;
               }
           }
           Err(SysError::IndexOutOfBound) => break,
           Err(_) => return ERROR_LOAD_LOCK_HASH,
       }
   }
   ```

   Note: `Source::GroupInput` already groups by lock script hash, so all cells share the same lock hash. This explicit args check is defense in depth per Officeyutong's recommendation.

   Actually, `Source::GroupInput` groups by the FULL lock script (code_hash + hash_type + args). So all group inputs by definition have identical lock args. The check is truly redundant but adds explicitness. Add it with a comment noting it's defense in depth.

   Simpler approach: just add a comment documenting that GroupInput guarantees identical lock args. Skip the runtime check since it's provably redundant (GroupInput matching guarantees it). Per Officeyutong's concern: the issue might be that `validate_merge` uses `Source::Output` with manual lock hash comparison for the output, not the inputs. The inputs are fine (GroupInput). The output check is already done via `our_lock_hash` comparison. This is sufficient.

   **Decision:** Add the explicit lock args check on the merge output cell (not inputs — inputs are already guaranteed by GroupInput). Verify the output cell's full lock args match our lock args, not just the hash:

   // (defense in depth — hash collision is astronomically unlikely but explicit is better)
