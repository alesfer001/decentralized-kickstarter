**Attack 1 — Fail-safe backdoor (Issue 1):**
- Create campaign, pledge, finalize as Success
- Attempt refund WITHOUT campaign cell_dep (omit it from transaction)
- Expected: Transaction rejected with ERROR_CAMPAIGN_CELL_DEP_MISSING

**Attack 2 — Inflated receipt (Issue 2):**
- Create campaign, pledge 100 CKB
- Attempt to create receipt with pledge_amount = 1000 CKB (10x inflation)
- Expected: Transaction rejected with ERROR_AMOUNT_MISMATCH

**Attack 3 — Arbitrary partial refund (Issue 3):**
- Create merged pledge (200 CKB from 2 backers)
- Attempt partial refund of 150 CKB when receipt says 100 CKB
- Expected: Transaction rejected with ERROR_REFUND_AMOUNT_MISMATCH

**Attack 4 — Campaign destruction on Success (Issue 1b):**
- Create campaign, pledge, finalize as Success
- Attempt to destroy campaign cell (no type output)
- Expected: Transaction rejected with ERROR_DESTRUCTION_NOT_ALLOWED

**Attack 5 — Premature finalization (Issue 5):**
- Create campaign with future deadline
- Attempt finalization with since < deadline
- Expected: Transaction rejected (by both campaign-lock and campaign type script)

These tests validate that the security fixes actually block the attack vectors. If any attack succeeds, the corresponding fix needs correction.
